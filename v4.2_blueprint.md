# PhantomClaw v4.2 — Observability & Operational Truth Blueprint

> Purpose: make the EA–Bridge–Bot–AI–Dashboard loop operationally transparent, measurable, and debuggable in real time.
> Focus: visibility and reliability signals, not strategy/risk logic changes.

---

## Why v4.2

v4.1 improved hardening, dashboard UX, and model controls.  
The remaining gap is **operator confidence**:

1. Is EA truly connected and healthy now?
2. Is auth valid (token + contract) now?
3. Is AI reachable and producing decisions now?
4. Is dashboard live data fresh or stale?
5. Is end-to-end flow healthy or silently degraded?

This blueprint introduces a single source of operational truth.

---

## Problem Statement

System behavior can be correct while operators still feel blind.

Current pain points:

1. EA lacks explicit on-chart status for bridge/auth/contract/decision loop health.
2. No canonical end-to-end health model (`GREEN/AMBER/RED`) with reason codes.
3. Dashboard shows data but does not clearly express freshness contracts per panel.
4. No single endpoint that answers: "Is the bot safe/healthy to run now?"

---

## Scope

### In Scope

1. Canonical `SystemTruth` status model for EA, bridge, AI, decision loop, dashboard sync.
2. Standard reason codes and freshness/latency metrics.
3. Shared status surfaces:
   1. EA on-chart panel
   2. Dashboard Operations strip
   3. Telegram status summary
4. Runbook mappings from reason code -> operator action.

### Out of Scope

1. Strategy generation logic changes.
2. Risk formula/guardrail changes.
3. Trade execution model redesign.

## Architecture: Current vs Target

```text
CURRENT (v4.1)
EA -> /signal -> Bridge -> async AI decision -> queue -> EA /decision -> execution -> /trade-result
Dashboard reads runtime + DB + logs (SSE + fallback polling)

TARGET (v4.2)
Same trading flow, plus:
- Canonical health state (SystemTruth)
- Measured latency/freshness across every link
- Unified status in EA, Dashboard, Telegram
- Clear RED/AMBER/GREEN with reason codes
```

Key point: architecture is medium-complex but valid.  
v4.2 addresses **observability debt**, not structural rewrite.

## Operational Truth Model

Define canonical object: `SystemTruth`.

Top-level sections:

1. `ea_link`
2. `bridge_auth`
3. `contract_compat`
4. `decision_loop`
5. `ai_health`
6. `dashboard_sync`
7. `data_freshness`
8. `overall`

Each section must include:

1. `status`: `GREEN | AMBER | RED`
2. `reason_code`: stable machine-readable key
3. `message`: human-readable summary
4. `last_ok_at`: RFC3339 UTC
5. `last_error_at`: RFC3339 UTC (if any)
6. `metrics`: key-value map

---

## Status Semantics (Initial Thresholds)

| Domain | GREEN | AMBER | RED |
|---|---|---|---|
| EA Link | last signal age <= 2x signal interval | >2x and <=5x | >5x or never seen |
| Bridge Auth | recent auth checks pass | intermittent unauthorized | persistent unauthorized |
| Contract | major version aligned | occasional mismatch | persistent mismatch |
| Decision Loop | ack->decision p95 < 5s, consume healthy | p95 5-15s or consume lag | no decisions/queue stuck/timeouts |
| AI Health | active provider healthy | fallback active/error burst | all providers failing or auth invalid |
| Dashboard Sync | SSE connected + fresh | polling fallback active | stale beyond threshold |

All thresholds should be config-backed later (not hardcoded long-term).

## Source-of-Truth Mapping

| Signal | Source | Notes |
|---|---|---|
| Runtime mode/session/risk | Bot runtime state | Real-time |
| Decisions/equity/analytics | SQLite | Durable/historical |
| Logs | structured log file | Operational trace |
| EA connectivity | bridge last signal timestamp | Derived from `/signal` traffic |
| Auth validity | bridge 200/401 telemetry | Token correctness |
| Contract validity | bridge contract checks | Header compatibility |
| AI health | router/provider status + errors | Includes fallback behavior |
| Dashboard liveness | SSE heartbeat + freshness ages | Must show stale state explicitly |

## Complexity / Redundancy / Clash Assessment

### Is the design too complex?
No. It is moderate complexity, appropriate for async trading.

### Is redundancy intentional?
Yes:

1. In-memory queue for speed
2. SQLite queue for durability
3. runtime + historical views for operations and analysis

### Actual clash risks

1. Mixed `request_id` and symbol fallback can blur correlation clarity.
2. Dashboard can appear "live" while portions are stale.
3. Auth/contract errors are not yet first-class operational states.

### v4.2 mitigation

1. Promote request correlation visibility
2. Show freshness and source metadata everywhere
3. Surface auth/contract failures at top-level health

1. Operator sees link/auth/AI/freshness in one glance

## Phase D3 — Hardening & Runbooks

1. Alert thresholds and debounce rules
2. Runbook mapping reason codes to actions
3. Failure-injection soak tests

Exit criteria:

1. Deterministic diagnosis paths
2. Actionable alerts with low noise

## Acceptance Criteria

v4.2 is complete when:

1. Auth mismatch is detectable within 10 seconds in both EA and dashboard.
2. Operator can distinguish EA-link delay vs AI-processing delay.
3. Dashboard proves freshness for each major panel.
4. Any stale/dead critical path raises `RED` with reason code.
5. "Is it safe to trade now?" can be answered in under 15 seconds.

## Operator Runbook (High-Level)

If `overall = RED`:

1. Check `bridge_auth.reason_code`
2. Check `ea_link.last_signal_age_sec`
3. Check `ai_health` provider errors/fallback state
4. Check queue depth and oldest pending age
5. Decide mode:
   1. continue (if false alarm)
   2. observe-only
   3. halt
